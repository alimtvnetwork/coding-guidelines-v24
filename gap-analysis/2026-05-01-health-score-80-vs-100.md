## TL;DR

The score is **stuck at 80/100 by design, not by defect.**

- **Raw score:** 80/100 (B) — derived from `spec/health-dashboard.md`, which deducts **20 points** for **32 broken cross-folder links**.
- **Effective score after waiver:** **100/100 (A+)**.
- **All 32 deductions** point to **gitmap-v3 sibling repositories** (`01-app/`, `02-app-issues/`, `03-general/`, `12-consolidated-guidelines/`) that are **deliberately not synced** into this repo per the locked memory rule [`mem://constraints/avoid-app-sync`](mem://constraints/avoid-app-sync).
- The umbrella site **does** resolve those links; this repo, in isolation, never can.
- The folder-ref linter (`check-spec-folder-refs.py`) **passes** because every broken target is covered by [`spec-folder-refs.allowlist`](../linter-scripts/spec-folder-refs.allowlist) (25 allowlisted external folders).

So the 20-point gap is **not unfixed work**. It is a **classification artifact**: the dashboard subtracts unconditionally, while the audit guard correctly waives the same deductions.

## 3. Why these references exist (root cause, not a bug)

This repo (`coding-guidelines-v24`) is **one node** in a multi-repo umbrella. The spec files were originally authored against the umbrella's combined tree, where `../01-app/` resolves. When extracted into this standalone repo:

1. The siblings stay external (by policy).
2. The relative links remain in the source so the umbrella site keeps working.
3. The folder-ref linter was given an **allowlist** so CI doesn't fail.
4. The dashboard kept the **raw broken-link count** for transparency, then added an **"Effective (Waived) Score" row** explaining the waiver.

The badge happens to render the **raw** number, not the **effective** one.

The blind AI audit at **99.8/100** is the strongest external signal that the **content** is healthy. The 80 is purely a **link-resolution** artifact.
